Overview
In PA1, you extended xv6 by implementing new system calls and maintaining per-process system call
accounting. In PA2, you redesigned the xv6 scheduler and implemented a System-Call-Aware MultiLevel Feedback Queue (SC-MLFQ) scheduler. In PA3, you extended xv6’s virtual memory subsystem
by implementing page replacement and a swap mechanism using an in-memory array.
In this assignment, you will extend xv6 further by replacing the in-memory swap mechanism with a
disk-backed swap system, and by implementing disk scheduling policies and RAID-based storage.
Instead of storing swapped pages in memory, the kernel must now store them on disk. All swap-in and
swap-out operations must go through a disk layer that supports scheduling and RAID mapping.

Disk Latency Model
To make scheduling meaningful, you must simulate disk latency.
The disk is modeled as a sequence of logical blocks.
For each request, the latency must be computed as:
latency = |current_block - requested_block| + C
where:
•

current_block is the current disk head position

•

requested_block is the target block

•

C is a constant representing rotational delay

The kernel must maintain the current disk head position and update it after each request.

RAID-backed Swap
You must simulate multiple disks in software and distribute swap data across them using RAID
techniques.
RAID Configuration
The system must simulate 4 disks. All disks are identical and have equal capacity.
RAID 0 (Striping)
Data must be distributed across N disks. Logical block b must map to:
disk = b % N
block = b / N
RAID 1 (Mirroring)
Each block must be written to 2 disks. Reads may be served from any disk.
RAID 5 (Striping with Parity)
Data must be striped across N disks along with parity blocks. Parity must be computed using XOR
over data blocks. Parity must be distributed across disks using the rule:
parity_disk = block_number % N
On read:
•

data must be reconstructed if one disk is unavailable or data is missing

On write:
•

parity must be updated correctly

Swap and RAID Interaction
When a page is swapped out:
•
•

the page must be written to disk using the RAID mapping
multiple disk requests may be generated

When a page is swapped in:
•
•

the page must be read from the appropriate disk(s)
data must be reconstructed when required (RAID 5)
